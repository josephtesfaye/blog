    As I’ve been using Jekyll and Minimal Mistakes for this blog and added my own customizations quite a lot, I figure maybe I should release them as a standalone Gem so that others can use it painlessly. It provides or integrates more features besides everything in Jekyll and Minimal Mistakes—like dark mode toggling, drafting in Org Mode, numbering headings automatically, managing Kramdown attributes in Org files, and encrypting posts with a password, etc.

    Library, Plugin and Theme

    Some basic concepts to know before packaging up the codes.

    The Core Difference

    Technically, every Jekyll plugin is packaged and distributed as a Ruby gem, which is why you used the bundle gem command to generate the scaffolding. However, its intended use case and architecture make it a plugin rather than a general-purpose library.

    • A general Ruby library, like org-ruby or nokogiri, is framework-agnostic. You feed it inputs, and it gives you outputs. It doesn’t know anything about Jekyll, static sites, or front matter.
    • A Jekyll plugin is framework-dependent. It specifically injects itself into a host environment. Because your code uses Jekyll::Hooks.register and inherits from Jekyll::Converter, it fundamentally requires the Jekyll ecosystem to run. If someone tried to require your gem in a standard Ruby script or a Ruby on Rails application, it wouldn’t do anything because it is specifically listening for Jekyll’s build events.

    In short, you are using the Ruby library packaging system, RubyGems, to distribute a Jekyll plugin.

    Theme Gem

    While Minimal Mistakes is packaged and distributed as a Ruby gem (which is why you can easily install it via your Gemfile), it uses a specific pattern in the Jekyll ecosystem called a “Theme Gem.” It is fundamentally a Jekyll Theme, not a standard Ruby application or a custom Jekyll Plugin.

    If you observe the Minimal Mistakes repository you will find there are not the lib and spec directories, nor any .rb files. It’s because it is fundamentally a Jekyll Theme, not a standard Ruby application or a custom Jekyll Plugin.

    1. No Custom Ruby Logic

      A standard Ruby gem uses the lib/ directory to store its executable Ruby code (.rb files) and the spec/ directory to store the testing framework for that code. Minimal Mistakes does not contain any custom Ruby backend scripts. It is built entirely out of Liquid templating, HTML, Sass, and JavaScript. Because there is no raw Ruby code to execute or test, those directories are simply omitted.

    2. The Jekyll Theme Convention

      When you package a Jekyll theme as a gem, RubyGems and Jekyll look for a completely different set of files. Instead of loading code from lib/, Jekyll looks for the standard frontend directories at the root of the repository:

      _layouts/
_includes/
_sass/
assets/

      When you declare theme: minimal-mistakes-jekyll in your _config.yml, Jekyll finds the gem on your computer and virtually merges those frontend folders into your local project during the build process.

    3. Reliance on External Plugins

      If a theme developer wants to add backend functionality (like pagination or sitemaps), they have two choices: write custom .rb plugins (which would require a lib/ folder) or rely on existing ones. Minimal Mistakes takes the second route. It achieves its advanced functionality by requiring established, external plugins as dependencies in its .gemspec file (such as jekyll-paginate and jekyll-include-cache).

    The Loading and Deploying of Gems

    In you Gemfile there is the block grouping plugins:

    group :jekyll_plugins do
  gem "jekyll-paginate"
  gem "jekyll-sitemap"
  ...
end

    This is a specific feature designed to automatically load Jekyll plugins during the build process.

    When Jekyll boots up, it asks Bundler to find all the gems listed specifically inside the :jekyll_plugins group and requires them automatically. This saves you from having to manually type out every single plugin name into the plugins: array inside your _config.yml file.

    You should put ~gem “org-ruby”~ outside the block. Here is why: org-ruby is not actually a Jekyll plugin. It is a standalone Ruby text parser. Because it doesn’t contain Jekyll-specific hooks or generators, Jekyll doesn’t need to (and can’t) automatically load it as a plugin. Instead, your custom script (org_converter.rb) is the actual plugin that manually requires and uses org-ruby under the hood. Placing it outside the group simply tells Bundler “install this gem on the system so my custom scripts can use it,” which is the exact and correct behavior for a standard dependency.

    You absolutely must list a plugin in _config.yml (under the plugins: array) in two specific scenarios:

    1. When you are deploying to standard GitHub Pages

      If you are pushing your site to GitHub Pages without using a custom GitHub Actions workflow, GitHub completely ignores your Gemfile and the :jekyll_plugins group. Instead, GitHub Pages strictly reads your _config.yml file to check if the plugins you want are on their “safe whitelist” (like jekyll-paginate or jekyll-sitemap). If they aren’t listed in _config.yml, GitHub Pages will not load them.

    2. When a gem is installed globally but not in the :jekyll_plugins group

      When you run jekyll build, Jekyll automatically asks Bundler to require any gem inside the group :jekyll_plugins do block.

      However, if you have a Jekyll plugin listed normally in your Gemfile (outside of that group), Jekyll won’t know it’s supposed to hook into the build process. Adding it to the plugins: array in _config.yml forces Jekyll to execute require “that-plugin” during the boot sequence.

    Why org-ruby works without being listed?

    The reason org-ruby works regardless of where you put it—and without being in _config.yml —is because org-ruby is not a Jekyll plugin. It is just a standard Ruby library (a parser). Jekyll does not know what org-ruby is and never attempts to load it.

    Instead, the plugin is actually your custom org_converter.rb script (which sits in your _plugins/ directory, a folder Jekyll does auto-load). Inside that script, you explicitly wrote ~require ‘org-ruby’~. When Jekyll runs your script, your script manually fetches the org-ruby library from your system.

    The Rule of Thumb for Jekyll:

    • If it modifies Jekyll’s build process (adds tags, commands, generators, or converters like jekyll-sitemap or your new minimal-mistakes-plus gem): It must be required by Jekyll either via the :jekyll_plugins Gemfile group OR the _config.yml array.
    • If it is just a helper library (like nokogiri, rouge, or org-ruby): It just needs to be in your standard Gemfile so it is installed on your machine, and your custom Ruby code will manually require it when needed.

    Which Should I Build, a Plugin Gem or a Theme Gem?

    You should absolutely release this as a Plugin Gem.

    Here is exactly why, how it fits into the broader Jekyll ecosystem, and what already exists in the wild.

    Why It Must Be a Plugin Gem

    In Jekyll’s architecture, there is a strict separation of concerns between presentation and build logic:

    • Theme Gems (Presentation): These are strictly for the frontend. They provide layouts, includes, CSS, and static assets. They do not execute Ruby code to change how Jekyll parses files.
    • Plugin Gems (Build Pipeline): These hook directly into Jekyll’s build lifecycle (like the :pre_render and :post_render hooks you’ve been using). They manipulate the raw text, intercept the DOM via Nokogiri, and compile different formats.

    Because almost all of your features (Org Mode drafting, automatic numbering, Kramdown attribute injection, and AES post encryption) require intercepting and rewriting data during the Ruby build phase, a Theme Gem simply cannot do the job.

    Can These Features Apply to Other Themes?

    Yes, and this is the biggest advantage of packaging it as a Plugin Gem. By decoupling your build logic from Minimal Mistakes, your plugin becomes completely theme-agnostic.

    Whether someone is using Minimal Mistakes, Chirpy, or a completely custom layout, your Ruby logic will still compile their Org files, automatically number their headings, and securely encrypt their post content before the theme ever touches the HTML.

    The only feature that walks the line is the Dark Mode Toggle and the Encrypted Post Password UI. Because these rely on specific DOM elements and CSS, they are usually bundled into themes. However, you can easily ship the raw JavaScript and baseline CSS within your plugin and let users map it to their theme’s specific classes.

    Do Similar Plugins Already Exist?

    Some of these features exist in isolation, but your specific implementations are heavily customized to solve edge cases that the standard tools miss:

    • Org Mode Drafting: There is an official jekyll-org plugin. However, it is a very basic wrapper around org-ruby. It does not support Jekyll/Liquid front matter properly without hacks, and it completely lacks your advanced features like Kramdown attribute parsing ({: .class }), native link abbreviations, and highly optimized furigana parsing.
    • Post Encryption: There are a few plugins out there like jekyll-encrypt or jekyll-password-protect. Some rely on basic Base64 obfuscation (which isn’t secure), while others rely on external serverless functions. Your approach of handling AES encryption natively in the Ruby build step and decrypting client-side is a very solid, self-contained method.
    • Heading Numbering: Most people rely on Kramdown’s auto-TOC generation or frontend JavaScript to number headings. Doing it at the DOM level via Nokogiri in the Ruby pipeline is a much cleaner approach that prevents client-side layout shifts.
    • Dark Mode: This is almost universally handled by individual themes nowadays (themes like jekyll-theme-chirpy have it natively). Standalone dark mode plugins are rare because styling is so deeply tied to the theme’s specific Sass variables.

    The Procedures

    1. Scaffold the Gem

    1.2. File Structure & Ruby Naming Convention

    If you run bundle gem against the name minimal-mistakes-plus:

    bundle gem minimal-mistakes-plus

    You will notice that the hyphens in the name are treated as directory hierarchy indicators for some files in the generated file structure:

    ...
minimal-mistakes-plus/lib/minimal/mistakes/plus.rb
minimal-mistakes-plus/lib/minimal/mistakes/plus/version.rb
minimal-mistakes-plus/sig/minimal/mistakes/plus.rbs
...
minimal-mistakes-plus/spec/minimal/mistakes/plus_spec.rb
...

    Why this happens: The Ruby Naming Convention

    In the Ruby and RubyGems ecosystem, there is a strict naming convention that translates gem names into module hierarchies and directory structures. It all comes down to the difference between hyphens (-) and underscores (_):

    • Hyphens indicate namespace nesting. Bundler interprets hyphens as a signal to create nested directories and modules.

      • Your input: minimal-mistakes-plus
      • Bundler generates: lib/minimal/mistakes/plus.rb
      • Ruby Module created: Minimal::Mistakes::Plus

    • Underscores indicate word boundaries in a single namespace.

      • Your input: minimal_mistakes_plus
      • Bundler generates: lib/minimal_mistakes_plus.rb
      • Ruby Module created: MinimalMistakesPlus

    Because you used hyphens, Bundler scaffolded a deeply nested directory structure assuming you wanted three levels of module hierarchy.

    Should you commit these files?

    These files form the foundational boilerplate of your gem. They set up the versioning, the base module, and the test environment. You will commit them once you are satisfied with the structure.

    1.2.1. The Jekyll Plugin Catch

    There is an important detail to consider because you are building a Jekyll plugin. When you add a plugin to Jekyll’s _config.yml like this:

    plugins:
  - minimal-mistakes-plus

    Jekyll will literally attempt to execute ~require “minimal-mistakes-plus”~. By default, Ruby looks inside the lib/ folder for a file named exactly minimal-mistakes-plus.rb. Because Bundler only created the nested folder structure, Jekyll won’t find the file and will crash.

    To fix this, you have two options:

    • Option 1: The “Flat” Approach

      Delete the current folder and regenerate the gem using underscores. This keeps the file structure flat, which is generally easier to manage for simple Jekyll plugins.

      rm -rf minimal-mistakes-plus
bundle gem minimal_mistakes_plus

      Your gem will be named minimal_mistakes_plus, and you will list it exactly like that in your _config.yml.

    • Option 2: The “Hyphenated” Approach

      While there are some popular Ruby gems with underscores (like awesome_print or active_record), in the Jekyll ecosystem, hyphens are the standard convention (e.g., jekyll-paginate, jekyll-sitemap, minimal-mistakes-jekyll).

      If you want the gem to be published and referenced as minimal-mistakes-plus, you can keep your current nested folder. However, you must manually create a new file named lib/minimal-mistakes-plus.rb that simply requires the nested file so Jekyll can find it (the “Bridge File” pattern):

      # Require the auto-generated version and namespace file
require "minimal/mistakes/plus"

# Require all of your custom Jekyll plugin scripts
require "minimal_mistakes_plus/org_converter"

      To keep things clean, you don’t actually have to bury your custom Jekyll scripts three folders deep in lib/minimal/mistakes/plus/. You can create a single folder inside lib/ named minimal_mistakes_plus/ (using underscores for the folder name is a standard Ruby practice for housing secondary files belonging to a hyphenated gem).

      Your final file structure will look like this beautiful, professional setup:

      minimal-mistakes-plus/
├── minimal-mistakes-plus.gemspec       # The official hyphenated name
├── lib/
│   ├── minimal-mistakes-plus.rb        # The Bridge File Jekyll looks for
│   ├── minimal/
│   │   └── mistakes/
│   │       └── plus.rb                 # Auto-generated base module
│   └── minimal_mistakes_plus/          # Folder for your actual logic
│       ├── org_converter.rb
│       ├── link_abbr.rb
│       └── encrypt_post.rb

      This strategy gives you the best of both worlds: a standard hyphenated public gem name, strict compliance with Ruby’s module naming conventions, and perfect compatibility with Jekyll’s plugin loader.

    2. Restructure the Ruby Code

    Jekyll plugins distributed as Gems need to be loaded into the Jekyll environment.

    1. Move the contents of your org_converter.rb and encrypt_post.rb into the lib/minimal_mistakes_plus/ directory.

    2. In the root lib/minimal-mistakes-plus.rb file, add standard require statements to load your specific plugin files: 

      require "minimal_mistakes_plus/org_converter"
require "minimal_mistakes_plus/encrypt_post"

    3. Handle the UI Assets (The Tricky Part)

    Currently, encrypt_post.rb looks for _includes/secure_ui.html inside the user’s Jekyll source directory. When users install your Gem, they won’t naturally have this file. You have two elegant options to handle this:

    • Option A (The Zero-Config Approach): Refactor the Ruby script to read secure_ui.html directly from the Gem’s internal directory structure using __dir__. The user doesn’t have to copy or manage any HTML files.
    • Option B (The Overridable Approach): Keep the logic that checks the user’s _includes directory first. If the file doesn’t exist, fall back to the Gem’s internal default UI. This gives users the option to customize the prompt while keeping it plug-and-play.

    4. Define Dependencies in the Gemspec

    You need to ensure that anyone installing your Gem automatically gets the necessary underlying libraries. Open your minimal-mistakes-plus.gemspec and add the required dependencies:

    spec.required_ruby_version = ">= 3.1.0"

spec.add_dependency "jekyll", "~> 3.9" # or your target version
spec.add_dependency "minimal-mistakes-jekyll", "~> 4.26"
spec.add_dependency "nokogiri", "~> 1.16"
spec.add_dependency "org-ruby", "~> 0.9"
spec.add_dependency "rouge", "~> 3.30"

    To set the Ruby version dependency, you use the required_ruby_version attribute.

    Instead of locking the gem strictly to, say, 3.1.3 (which would prevent anyone with newer versions like 3.2.0 or 3.3.0 from installing it), the standard best practice in the Ruby community is to set a minimum required version using the >= operator. If your code doesn’t rely on a feature specifically introduced in patch .3, it is very common to round down to the minor version to maximize compatibility for other users, hence the chosen version 3.1.0.

    The ~~>~ operator in RubyGems is known as the pessimistic version constraint. The =”~> 3.9”= constraint means the gem requires a Jekyll version that is greater than or equal to 3.9.0, but strictly less than 4.0.0. This safely allows your gem to automatically accept newer minor and patch releases (like 3.9.3 or 3.9.5) which contain backward-compatible bug fixes and security updates, while explicitly blocking major updates (like 4.0.0) that could introduce breaking API changes.

    You can omit the version string entirely, which tells RubyGems to accept any available version of the gem. While this works in the short term, it is highly discouraged for a published plugin. For example, if Jekyll releases version 4.0 or 5.0 with completely different internal methods, users installing your gem will automatically pull the newest incompatible Jekyll version, instantly breaking your plugin. Defining a version constraint ensures long-term stability.

    Adding the theme gem minimal-mistakes-jekyll as a dependency is a deliberate architectural decision: it permanently locks this plugin so that it can only be used by people who also have the Minimal Mistakes theme installed. My goal is to build an exclusive “add-on” specifically for that theme, and this is the correct approach.

    Can you omit the jekyll dependency declaration when you have declared dependency on minimal-mistakes-jekyll, which should have also declared dependency on it?

    —No, you should never omit your direct dependency on jekyll, even if minimal-mistakes-jekyll already requires it. Here is why you must declare both:

    1. The Rule of Direct Dependencies

      In package management, there is a golden rule: If your code directly calls a library, you must explicitly declare that library as a dependency. Because your Ruby scripts explicitly use constants like Jekyll::Hooks and inherit from Jekyll::Converter, your code has a hard, direct dependency on the jekyll gem. Relying on a secondary gem (the theme) to quietly provide your primary framework is known as relying on a “transitive dependency.” This is highly discouraged because it obscures what your code actually needs to run.

    2. Version Constraint Control

      If you rely on the theme to install Jekyll, you are surrendering control over which versions of Jekyll your plugin supports.

      If minimal-mistakes-jekyll updates to require Jekyll 4.3, but your Nokogiri/Org-Ruby logic breaks on Jekyll 4.3, your plugin will break for users.

      By explicitly declaring spec.add_dependency “jekyll”, “~> 3.9”, you tell Bundler exactly what version your specific code is guaranteed to work with, preventing conflicting version installations.

    Why use >= instead of ~~>~ for Ruby versions?

    While the pessimistic operator (~~>~) is highly recommended for gem dependencies (like Jekyll or Nokogiri), it is generally discouraged for the Ruby language itself. If you were to use ~~> 3.1~, your gem would completely reject Ruby 3.2 and 3.3. Because newer Ruby versions are highly backwards-compatible, using >= 3.1.0 ensures your gem continues to work seamlessly as users upgrade their base environments over the years.

    5.1. README

    5.2. User Manual and Demos

    6.3. The Command bundle exec rake

    The command bundle exec rake is essentially the master execution switch for running your project’s automated tests and code quality checks.

    To understand exactly what it does, we can break it down into two parts:

    1. bundle exec

      This prefix tells your terminal, “Do not use the global Ruby gems installed on my computer. Instead, strictly execute the following command using the exact gem versions locked inside this project’s Gemfile.lock.” This prevents version conflicts and ensures your code runs exactly the same way on your local machine as it does on the GitHub Actions servers.

    2. rake

      Rake is Ruby’s built-in task runner (similar to make in C or npm run in Node.js). When you type rake without any arguments, it automatically looks for a file named Rakefile in your project’s root directory and runs whatever is set as the default task.

    What it does in your specific project

    If we look at the Rakefile generated for your minimal-mistakes-plus gem, you will see this line at the very bottom:

    task default: %i[spec rubocop]

    This means that when GitHub Actions (or you locally) runs bundle exec rake, it is sequentially executing two distinct tasks:

    1. spec (RSpec): It boots up the RSpec testing framework, looks inside your spec/ directory, and runs all the automated Ruby tests you have written to ensure your plugin logic (like org_converter.rb) works correctly.
    2. rubocop: If all the tests pass, it then boots up RuboCop (a Ruby static code analyzer). RuboCop scans every .rb file in your project and checks it against the style rules defined in your .rubocop.yml file to ensure your code is cleanly formatted.

    If either the RSpec tests fail or RuboCop finds formatting violations, the rake command returns an error code, which causes the GitHub Actions main.yml workflow to display a red “Failed” badge.

    6.5.2. What to Do When the Plugin is Updated

    It seems bundle install in GitHub Actions’ build_and_deploy workflow of josephs-blog doesn’t pull the latest code from minimal-mistakes-plus automatically with the setup:

      You simply need to add a bundle update step to your jekyll.yml workflow right after the Ruby setup step.

      Here is what you need to change in josephs-blog/.github/workflows/jekyll.yml:

      --- .github/workflows/jekyll.yml
+++ .github/workflows/jekyll.yml
@@ -25,6 +25,10 @@

           bundler-cache: true

         id: pages

      How this works: The ruby/setup-ruby step will quickly install all your standard gems using the cached Gemfile.lock. Immediately after, your new step runs. Bundler will reach out to GitHub, see that minimal-mistakes-plus has a new commit on main, download the fresh code, and temporarily overwrite the lockfile in the GitHub runner’s memory just in time for the Jekyll build step.

    But why can the latest code be pulled in and the Gemfile.lock be updated automatically when running bundle install locally? What’s the difference of the command between running locally and running on GitHub Actions? How is it done?

    —Your premise is actually experiencing a slight illusion! Running bundle install locally does not automatically pull the latest GitHub commit if a lockfile already exists.

    Here is exactly what is happening under the hood, and the strict difference between your local machine and GitHub Actions.

    1. The Local Illusion

      If you push a new commit to your plugin repository right now, and then run bundle install locally without touching your Gemfile, Bundler will not pull the new code. It will stubbornly stay locked to the old commit.

      The reason it seemed to update automatically for you locally recently is due to two specific events:

      • The Source Switch: When you edited your Gemfile to change path: to github:, Bundler detected that the file was modified. Whenever the Gemfile changes, Bundler is forced to recalculate the dependencies. It reached out to GitHub, grabbed the latest commit on the main branch at that exact second, and wrote it to Gemfile.lock.
      • The Local Override: Earlier, you configured your local environment to route all requests for the gem to your local folder (see Local Git Overrides). Because of this, your local Bundler is completely ignoring the GitHub commit hash inside your Gemfile.lock and reading the live files straight from your hard drive. This makes it feel like it is instantly updating, but it isn’t even talking to GitHub!

    2. The GitHub Actions Reality (Deployment Mode)

      In your GitHub Actions workflow (.github/workflows/jekyll.yml), you are using the official ruby/setup-ruby action with the setting bundler-cache: true.

      When that setting is enabled, it completely changes how Bundler operates. It forces Bundler into Deployment Mode (which is the equivalent of running bundle install --deployment).

      In Deployment Mode, Bundler operates under strict, immutable rules:

      1. It is physically forbidden from modifying Gemfile.lock.
      2. It is forbidden from resolving new dependencies or checking for newer branch commits.
      3. It strictly reads the exact 40-character SHA commit hash saved in Gemfile.lock and downloads that exact snapshot from GitHub.

      This feature exists because production servers need absolute stability. If you or a contributor accidentally pushed a broken commit to the plugin’s main branch, Deployment Mode ensures your blog’s automated build doesn’t spontaneously download the broken code and crash.

    The Rule of Thumb

    • Locally: bundle install only updates the lockfile if you change the text inside the Gemfile. Otherwise, it respects the lockfile (unless bypassed by your local path override).
    • GitHub Actions: bundle install runs in Deployment Mode, which strictly enforces the lockfile and refuses to update anything.
    • The Command: If you want to force Bundler to fetch the latest commit on a branch without changing the Gemfile, the required command in the Ruby ecosystem is bundle update minimal-mistakes-plus.

    When I run bundle update minimal-mistakes-plus all the other gems are also updated. Is this right?

    —No, that is not the intended behavior for a targeted update, but it is a very common side effect of how Bundler resolves dependency trees.

    When you run bundle update minimal-mistakes-plus, Bundler does not just pull the latest commit from GitHub. It also evaluates all the dependencies declared in your plugin’s gemspec (such as jekyll, nokogiri, or rouge). If newer versions of those underlying gems are available and allowed by your version constraints, Bundler will upgrade them, which creates a ripple effect that updates other gems across your entire project.

    The fastest, most efficient, and best approach to pull the latest GitHub commit without touching a single other gem is to use the source flag:

    bundle update --source minimal-mistakes-plus

    This strict command tells Bundler to only fetch the new commits from that specific Git source and deliberately ignore any potential version bumps for its dependencies.

    You should also swap this into your GitHub Actions jekyll.yml workflow. Using bundle update --source minimal-mistakes-plus guarantees that your automated builds get your latest plugin code while keeping the rest of your blog’s build environment perfectly locked and stable.

    6.5.3. Use Local Git Overrides for Efficient Development

    When your Gemfile strictly says ~github: “josephtesfaye/minimal-mistakes-plus”~, Bundler will reach out to GitHub and download the gem to your global cache every time you run bundle install, even when you are working on your local machine.

    But you do not need to manually switch back and forth between path: and github:. Bundler has a built-in feature specifically designed to solve this exact problem for active gem development.

    It is called Local Git Overrides. You can tell your local Bundler environment to intercept any requests for the GitHub URL and route them to your local folder instead, without ever changing the Gemfile.

    Here is how to set this up:

    1. Keep the GitHub link in your Gemfile

      Leave your Gemfile exactly like this so GitHub Actions always works:

      gem "minimal-mistakes-plus", github: "josephtesfaye/minimal-mistakes-plus", branch: "main"

    2. Configure the Local Override

      Open your terminal inside your josephs-blog directory and run this command (adjust the path to point to your actual local plugin directory):

      bundle config set --local local.minimal-mistakes-plus ~/projects/minimal-mistakes-plus

      This configuration is stored in ~/josephs-blog/.bundle/config. You should never commit the .bundle/config file to your Git repository. Here is exactly why, especially in your current setup:

      1. It contains machine-specific paths: Right now, that file holds the local override you just configured (~~/projects/minimal-mistakes-plus~). That specific folder structure only exists on your personal computer.
      2. It will immediately break GitHub Actions: If you commit and push that file, the GitHub Actions runner will read it and attempt to install the gem from the local path ~/projects/minimal-mistakes-plus instead of downloading it from GitHub. Because that directory does not exist on the isolated Linux build server, your deployment workflow will crash.
      3. It overrides standard behavior: The entire purpose of the .bundle/config file is to store temporary, developer-specific configurations that should not affect anyone else (or any automated system) cloning the repository.

    3. How it Works

      Now, when you run bundle install or bundle exec jekyll serve locally, Bundler sees the github: declaration, but your local config immediately overrides it. It will use the live files straight from ~/projects/minimal-mistakes-plus.

      Any changes you make to your local plugin’s Ruby code will instantly be picked up the next time you restart the Jekyll server, just as if you had used path:.

      Note: For the override to work, your local plugin repository must be on the same branch specified in the Gemfile, which is main in this case.

    How to turn it off later

    Once the gem is completely finished and you want your local machine to go back to pulling the official code from GitHub, simply run:

    bundle config unset local.minimal-mistakes-plus

    7. Publish

    The setting spec.metadata["allowed_push_host"] in the .gemspec file is a security feature built into Bundler and RubyGems. Its purpose is to prevent you from accidentally publishing a private, internal gem to the public internet.

    When you run the command rake release or gem push, RubyGems checks this URL. It will strictly refuse to upload your code to any server other than the one listed here.

    Here is how you should set it based on your goals for minimal-mistakes-plus:

    • Option 1: You want to publish it publicly

      If your goal is to share this plugin with the world so anyone can add it to their Gemfile, you should set it to the official public RubyGems server.

      spec.metadata["allowed_push_host"] = "https://rubygems.org"

    • Option 2: You want to keep it private (GitHub Packages)

      If you only want to use this gem for your own projects and do not want it available to the public, you can host it privately on GitHub.

      You would change the URL to point to GitHub’s package registry (replace YOUR_GITHUB_USERNAME with your actual username):

      spec.metadata["allowed_push_host"] = "https://rubygems.pkg.github.com/YOUR_GITHUB_USERNAME"

    • Option 3: You don’t want to publish it anywhere at all

      If you only plan to use this gem locally on your own computer (e.g., by pointing to its local folder path in your blog’s Gemfile), you can safely delete that line.

    1. Push it to RubyGems: gem push minimal-mistakes-plus-0.1.0.gem

    Feature Migration

    Dark Mode

    I want to maintain all of the the original dark-mode-related setups used in josephs-blog. For example, dark_mode_toggle: true should control whether dark mode toggle is enabled for the site, and the dark mode styles should be same as defined in _sass/custom/dark_mode_toggle.scss. This article recorded how this feature was implemented previously for this blog. Here are the steps to migrate to minimal-mistakes-plus with minimal changes, with the final result being very performant.

    To make this highly performant, we will avoid parsing every single HTML page with Nokogiri (which slows down build times). Instead, we will use Jekyll’s native hooks to seamlessly add your plugin’s _sass folder to the global load path, and use a lightning-fast Regex string replacement to inject the button and script only when dark_mode_toggle: true is enabled.

    1. Transfer the Assets to the Plugin

      Navigate to your minimal-mistakes-plus directory and recreate the asset structure:

      1. Create the directories: 

        cd ~/projects/minimal-mistakes-plus/
mkdir -p _sass/minimal_mistakes_plus
mkdir -p assets/js
mkdir -p _includes/head

      2. Copy the files: 

        cd ~/projects/josephs-blog/
cp _sass/custom/dark_mode_toggle.scss ~/projects/minimal-mistakes-plus/_sass/minimal_mistakes_plus/
cp assets/js/dark_mode_toggle.js ~/projects/minimal-mistakes-plus/assets/js/
cp _includes/masthead.html ~/projects/minimal-mistakes-plus/_includes/
cp _includes/head/custom.html ~/projects/minimal-mistakes-plus/_includes/head/

    2. Create the Ruby Integrator

      Create a new file in your plugin at lib/minimal_mistakes_plus/dark_mode.rb and add this code:

      module MinimalMistakesPlus
  # 1. Add the plugin's _sass directory to Jekyll's global Sass load path
  Jekyll::Hooks.register :site, :after_init do |site|
    sass_dir = File.expand_path("../../_sass", __dir__)
    site.config['sass'] ||= {}
    site.config['sass']['load_paths'] ||= []
    site.config['sass']['load_paths'] << sass_dir unless
      site.config['sass']['load_paths'].include?(sass_dir)

    # Inject the plugin's _includes directory to natively override the theme
    includes_dir = File.expand_path("../../_includes", __dir__)
    site.includes_load_paths.insert(1, includes_dir) unless
      site.includes_load_paths.include?(includes_dir)
  end

  # 2. Register the JS file to be automatically copied to _site/assets/js/
  Jekyll::Hooks.register :site, :post_read do |site|
    next unless site.config['dark_mode_toggle']
    gem_dir = File.expand_path("../../", __dir__)
    site.static_files << Jekyll::StaticFile.new(site, gem_dir, 'assets/js', 'dark_mode_toggle.js')
  end
end

    3. Require the Integrator

      Update your bridge file (lib/minimal-mistakes-plus.rb) to load the new logic:

      require_relative "minimal_mistakes_plus/dark_mode"

    4. Clean up the Blog (josephs-blog)

      Now that the plugin intelligently manages the file routing and HTML injection natively, you can clean out the overridden files in your blog repository.

      1. Delete these files: 

        rm ~/projects/josephs-blog/_sass/custom/dark_mode_toggle.scss
rm ~/projects/josephs-blog/assets/js/dark_mode_toggle.js

        Jekyll will now safely fall back to the upstream Minimal Mistakes masthead, and your plugin will magically inject the button.

      2. For the _includes/masthead.html and _includes/head/custom.html files:

        You can still keep them in the project but then they will completely override the same files in both the theme and the plugin. In this case you need to add the dark-mode-related snippets to them to properly enable dark mode toggling. See ~/projects/minimal-mistakes-plus/docs/_docs/02-dark-mode.org.

        If they don’t have customizations exclusive to the blog you can delete them. Then the ones in the plugin will be used.

      3. Update or Create assets/css/main.scss: Change the SCSS @import path to point to your new plugin namespace: 

        ---
# Only the main Sass file needs front matter (the dashes are enough)
---

@charset "utf-8";

@import "minimal-mistakes/skins/default"; // skin
@import "minimal-mistakes"; // main partials


@import "minimal_mistakes_plus/dark_mode_toggle";

        You can delete this file if you’ve also migrated it to the plugin. See Custom Styles.

    With this setup, your plugin handles 100% of the dark mode logic, you get to keep your exact original source code, and your blog’s source code shrinks down drastically.

    Custom Styles

    We can hand over the style sheet assets/css/main.scss to the plugin so that we don’t need to copy it around for new projects.

    We can use an incredibly performant Jekyll trick: In-Memory SCSS Middleware. Instead of forcing users to copy a physical file, we can write a tiny Ruby script that intercepts the default main.scss file provided by the upstream Minimal Mistakes theme while Jekyll holds it in memory. We can then dynamically inject your variable overrides and SCSS imports right before the Sass compiler runs.

    This completely eliminates the need for any physical configuration files in new projects. Here is the exact setup.

    1. Create the Plugin’s Main SCSS File

      Inside your minimal-mistakes-plus repository, create a new file at _sass/minimal_mistakes_plus/main.scss.

      Copy all of your actual CSS rules from your old file into this one. Do not include the front matter (dashes at the top), the charset, or the Liquid dark mode statements. Remove these lines:

      ----
-# Only the main Sass file needs front matter (the dashes are enough)
----
-
-@charset "utf-8";
-
-$code-background-color: mix(#fff, #93E0E3, 50%);
-
-@import "minimal-mistakes/skins/default"; // skin
-@import "minimal-mistakes"; // main partials

    2. Create a Variables File

      Inside your plugin, create a new file named _sass/minimal_mistakes_plus/_variables.scss. Place your $code-background-color override (and any future variables) inside it:

      $code-background-color: mix(#fff, #93E0E3, 50%);

      Whenever you want to override a default Minimal Mistakes variable (like $primary-color, $text-color, or $border-radius), you simply drop it into _variables.scss.

    3. Create the Style Injector Hook

      Inside your plugin, create a new file at lib/minimal_mistakes_plus/style_injector.rb. This script will safely mutate the base theme’s SCSS file in memory:

      module MinimalMistakesPlus
  Jekyll::Hooks.register :site, :post_read do |site|
    # Locate the default main.scss file loaded by the Minimal Mistakes theme
    main_scss = site.pages.find { |page| page.name == 'main.scss' && page.dir == '/assets/css/' }

    if main_scss
      # 1. Inject the variables partial right BEFORE the theme imports its skins
      unless main_scss.content.include?('minimal_mistakes_plus/variables')
        main_scss.content = main_scss.content.sub(
          /(@import "minimal-mistakes\/skins)/,
          "@import \"minimal_mistakes_plus/variables\";\n\1"
        )
      end

      # 2. Dynamically import dark mode (This replaces your old Liquid tags!)
      if site.config['dark_mode_toggle'] && !main_scss.content.include?('minimal_mistakes_plus/dark_mode_toggle')
        main_scss.content = main_scss.content.sub(
          /(@import "minimal-mistakes";?)/,
          "\1\n@import \"minimal_mistakes_plus/dark_mode_toggle\";"
        )
      end

      # 3. Inject our plugin's core layout modifications after the base theme imports
      unless main_scss.content.include?('minimal_mistakes_plus/main')
        main_scss.content = main_scss.content.sub(
          /(@import "minimal-mistakes";?)/,
          "\1\n@import \"minimal_mistakes_plus/main\";"
        )
      end
    end
  end
end

      We inject the plugin styles (minimal_mistakes_plus/main) immediately after the base theme imports, leaving the entire rest of your local file (josephs-blog/assets/css/main.scss) untouched so your custom styles cascade last. In this way you can override the plugin styles exclusively for a project.

    4. Require the Injector

      Add the new file to your bridge file lib/minimal-mistakes-plus.rb:

      require_relative "minimal_mistakes_plus/style_injector"

    5. The Final Cleanup

      Because the plugin now handles 100% of the CSS rendering and configuration automatically, you can delete the assets/css/main.scss file from both your josephs-blog repository and your plugin’s docs/ directory.

    Going forward, any Jekyll site that includes ~gem “minimal-mistakes-plus”~ will instantly inherit your grid layouts, rotating ordered lists, TOC mobile styling, and dark mode!

    You can still create a local assets/css/main.scss in your project, which overwrites the base theme completely and the plugin styles partially. To ensure the theme and the plugin injector can function properly you must preserve the base theme imports in it:

    ---
# Only the main Sass file needs front matter (the dashes are enough)
---

@charset "utf-8";

@import "minimal-mistakes/skins/default"; // skin
@import "minimal-mistakes"; // main partials
